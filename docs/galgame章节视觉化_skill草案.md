# 视觉化文字游戏章节

把指定章节改造成可游玩的动漫视觉小说，同时把剧情层、视觉层和展示层严格分开。优先保护原文、范围边界、可替换性和静态部署能力。

## 核心原则

1. 把现有剧情数据视为唯一权威来源。不得为适配画面而改写正文、对话、选项、条件、后果或顺序。
2. 把视觉配置和图像资产放在剧情文件之外。视觉调整不得要求同步编辑剧情文件。
3. 只处理用户指定的章节。不得顺手给其他章节添加视觉映射、分页、人物或背景。
4. 严格遵守原文已规定的人物、地点、道具、时间、关系和情绪；只补充原文没有规定的外貌与环境细节。
5. 让视觉层可以失败。配置或图片缺失时必须自动退回原有文字模式，剧情仍可完整游玩。
6. 使用相对路径和静态资源，不依赖后端、密钥或运行时图像 API，确保 GitHub Pages 可直接托管。

## 第一步：检查项目与锁定范围

在修改前完成以下检查：

- 定位仓库根目录、项目说明、`AGENTS.md`、剧情数据、入口页面、样式、播放器逻辑和测试。
- 查看工作区状态，记录已有未提交文件；不得覆盖或提交与任务无关的用户改动。
- 确认用户所说的“序章”或章节名称在数据中的准确 ID。
- 确认页面是否为纯静态结构，GitHub Pages 发布源位于哪个分支和目录。
- 记录指定章节之后的第一个章节，后续必须验证视觉模式在这里完全退出。

如果章节范围仍可从项目数据中可靠判断，直接采用最窄的合理范围并说明；只有不同解释会实质改变结果时才询问用户。

## 第二步：建立视觉事实表

完整阅读指定章节，包括：

- 所有开场场景；
- 所有可追问人物及回复；
- 所有决定选项；
- 每个决定对应的余波；
- 人物档案、延迟回声和与本章有关的全局设定。

为每个人物和场景区分两类信息：

### 原文硬约束

直接来自剧情，必须遵守。例如年龄线索、身份、服装、随身物、场所功能、时间、天气、人物当下态度和空间关系。

### 允许创作

原文没有规定、但生成画面必须补齐的信息。例如脸型、发型、室内材质、灯光方向、非品牌装饰和镜头构图。

不得把“允许创作”写成新的剧情事实。它只属于视觉设定。

创建一份章节视觉设定档案，至少包含：

- 统一画风；
- 成年人物比例与禁用风格；
- 每名角色的原文约束与允许补充；
- 每个背景的原文约束；
- 真实品牌、队徽、乱码文字和水印限制；
- 图像缺失时的回退规则。

## 第三步：设计独立视觉数据

新增单独的视觉配置文件，例如 `visual-data.json`。不要在其中复制剧情正文。

推荐结构：

```json
{
  "version": 1,
  "scope": ["chapter-id"],
  "backgrounds": {
    "room_morning": {
      "src": "assets/visuals/backgrounds/room-morning.webp",
      "alt": "上午的会议室"
    }
  },
  "characters": {
    "chair": {
      "name": "人物姓名",
      "expressions": {
        "neutral": "assets/visuals/characters/chair/neutral.webp",
        "stern": "assets/visuals/characters/chair/stern.webp"
      }
    }
  },
  "scenes": {
    "chapter-id.opening.0": {
      "background": "room_morning",
      "character": "chair",
      "position": "center",
      "beats": [
        { "expression": "neutral", "motion": "enter" },
        { "expression": "stern", "motion": "focus" }
      ]
    }
  }
}
```

遵守以下约束：

- `scope` 只能包含用户指定章节。
- 所有场景键必须以指定章节 ID 开头。
- 视觉场景只保存背景、人物、表情、位置、动画和色调。
- 文本始终从剧情数据读取。
- 一个剧情段落可以对应一个视觉 beat，但不得把段落内容复制到 beat 中。
- 同一余波包含不同人物时，允许每个 beat 覆盖人物与背景。

## 第四步：生成动漫素材

涉及新图像时，使用可用的图像生成能力，并遵守对应图像生成 skill 的工作流。优先使用当前可用的高质量图像模型；记录实际模型和最终提示词。

### 统一画风

使用类似以下基础方向，并按项目世界观调整：

> 高品质日系动漫视觉小说插画，清晰线稿，赛璐璐与柔和厚涂结合，电影化光影，当代职业环境，成年人真实年龄与职业比例，统一配色。避免校园萌系、幼态脸、Q版、真实品牌、水印和可读乱码。

### 背景

- 使用 16:9 横幅构图。
- 不放主要人物，避免人物与立绘重复。
- 为对话框和人物立绘预留安全空间。
- 不生成承载剧情信息的可读文字；重要信息必须留在正文中。
- 同一机构的多个房间保持色彩、建筑材料和规模一致。
- 先生成母版，再转成适合网页的 WebP。

### 人物立绘

- 为同一人物生成一张多表情设定母版，统一脸型、发型、服装、比例、视角和光线。
- 使用纯色抠图背景，优先选择不与人物服装冲突的纯洋红。
- 每名角色至少覆盖本章实际需要的中性、缓和与冲突表情，不为了数量虚构无用表情。
- 保持腰部以上或膝部以上的统一裁切，不让头发、手或道具被意外截断。
- 去除纯色背景后检查边缘、发丝、透明通道和服装颜色污染。
- 将每个表情切分为独立透明 WebP，并保留原始母版以便日后重做。

推荐目录：

```text
assets/visuals/
├── backgrounds/
├── characters/
│   └── character-id/
└── source/
```

### 图像复核

逐张检查：

- 是否为统一动漫风；
- 人物年龄、身份与服装是否符合原文；
- 场景是否符合时间和功能；
- 是否出现真实品牌、错误队徽、乱码、水印或多余人物；
- 同一人物不同表情是否保持身份一致；
- 透明立绘是否存在洋红残边；
- 网页成品尺寸和文件体积是否合理。

不合格时重新生成或编辑，不要用 CSS 掩盖明显的图像错误。

## 第五步：接入视觉舞台

在现有剧情卡片中增加一个独立视觉舞台，并保留原有文字容器。视觉舞台通常包含：

- 背景图；
- 渐变遮罩；
- 透明人物立绘；
- 地点标记；
- 章节视觉标记；
- 原有姓名、标题、正文、选项和按钮。

不要重新实现剧情系统。只在原播放器渲染当前场景时查询视觉映射。

推荐逻辑：

1. 正常加载剧情数据。
2. 单独尝试加载视觉配置；失败时记录警告并继续。
3. 只有当前章节属于 `scope` 且场景键存在时才显示视觉舞台。
4. 视觉场景存在多个 beats 时，按原正文段落索引切换表情或背景。
5. 最后一个 beat 完成后执行原按钮行为，不改变阶段推进逻辑。
6. 当前 beat 有人物时，姓名栏应与画面人物一致；无人物时保留剧情原姓名或旁白标记。
7. 离开指定章节时清除视觉分页状态，隐藏视觉舞台并恢复完整文字显示。

视觉状态可以加入存档，但必须使用可选字段并提供默认值，避免旧存档失效。

## 第六步：动画与响应式设计

只使用不改变剧情节奏的轻量动画，例如：

- 背景淡入；
- 人物从下方轻微进入；
- 聚焦时小幅前移；
- 紧张时极轻微横向位移。

避免循环晃动、强闪烁、长时间过场和强制等待。为 `prefers-reduced-motion` 禁用非必要动画。

所有视觉样式应限定在类似 `.visual-event` 的作用域内，防止影响其他章节。移动端至少验证：

- 人物不会挡住关键正文；
- 对话框可滚动或自然增高；
- 选择按钮可点击；
- 背景保持合理裁切；
- 姓名、地点和章节标记不重叠。

## 第七步：保证 GitHub Pages 兼容

- 使用相对路径，不以 `/` 开头，兼容 `用户名.github.io/仓库名/` 子目录。
- 不要求构建服务器、数据库、环境变量或密钥。
- 不在浏览器运行时调用图像生成接口。
- 保证 JSON、JavaScript、CSS、WebP 和 PNG 都由静态服务器提供。
- 如果项目没有构建步骤，让发布源直接指向默认分支根目录或实际站点目录。
- 更新项目说明，写明本地静态服务器和 Pages 发布方法。

## 第八步：验证

先做静态检查：

- JavaScript 语法检查通过。
- 原有剧情完整性测试通过。
- `story-data` 或等价剧情源没有 diff。
- 视觉配置 JSON 可解析。
- `scope` 只包含指定章节。
- 所有视觉场景键只属于指定章节。
- 所有背景和表情路径真实存在。
- HTML 中的新增元素 ID 与 JavaScript 引用一致。
- Git diff 没有空白错误。

再通过本地静态服务器实际试玩：

1. 从新游戏进入指定章节。
2. 检查每个开场 beat 的背景、人物、表情和原文。
3. 检查所有可追问人物的视觉映射。
4. 检查决定界面。
5. 至少覆盖每个决定余波的视觉映射；如果路线数量可控，全部覆盖。
6. 进入下一章节，确认视觉舞台隐藏、视觉 CSS 作用域退出、原有完整正文一次显示。
7. 检查浏览器控制台无错误。
8. 检查窄屏布局和减少动画模式。

如果用户要求上线：

1. 只暂存本次相关文件。
2. 提交前再次确认无关文件未被纳入。
3. 推送到实际 Pages 发布分支。
4. 等待 Pages 构建完成，并确认构建对应本次提交。
5. 在线检查首页、视觉 JSON 和至少一张背景及一张透明立绘返回成功。
6. 打开公开页面完成一次关键路径试玩。

## 自动测试建议

为视觉边界增加测试，至少断言：

```js
assert.deepEqual(visuals.scope, [targetChapterId]);
assert.ok(Object.keys(visuals.scenes).every((key) => key.startsWith(`${targetChapterId}.`)));

for (const assetPath of allVisualAssetPaths) {
  await access(new URL(assetPath, projectRoot));
}
```

同时保留原有全剧情路线测试，证明视觉化没有改变选择数量、后果和赛季结算。

## 禁止事项

- 不得把剧情正文复制进视觉配置。
- 不得为了图片方便而删减、润色或拆改原文。
- 不得用图像中的文字替代剧情信息。
- 不得视觉化未获授权的章节。
- 不得让图片加载失败阻断按钮或选择。
- 不得用人物好感度、属性条或“正确答案”提示改变原设计。
- 不得提交工作区中原有的无关文件。
- 不得把图像 API 密钥放进网页或仓库。
- 不得只检查文件存在而跳过实际试玩。

## 完成交付

最终向用户简明报告：

- 已视觉化的准确章节；
- 背景、人物和表情数量；
- 使用的图像模型；
- 剧情源是否保持未修改；
- 视觉配置和素材位置；
- 已运行的自动测试与试玩范围；
- GitHub Pages 状态与公开地址（如果已发布）；
- 保留但未触碰的无关文件。

同时保存最终提示词或提示词集合，使用户以后能独立重做背景与立绘。
